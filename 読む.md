それなら `plan` は「日付ごとのTODO」ではなく、**予定タイムブロック**として設計するのがよさそうです。

DBはこういう形が自然です。

```sql
create table planned_timeline_entries (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id),
  task_id uuid not null references tasks(id),

  planned_start_at timestamptz not null,
  planned_end_at timestamptz not null,

  status text not null default 'planned',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  check (planned_end_at > planned_start_at)
);
```

`planned_start_at` / `planned_end_at` は `date + time` ではなく `timestamptz` がいいです。  
理由は、timeline 側もたぶん実績の `start_at` / `end_at` を timestamp で扱うはずなので、同じ時間軸にそのまま載せられるからです。

実績との対応は、`timeline` 側に nullable FK を持たせるのが扱いやすいです。

```sql
alter table timeline_entries
add column planned_entry_id uuid null references planned_timeline_entries(id);
```

これで、

- 予定から開始した実績: `planned_entry_id` あり
- 突発作業の実績: `planned_entry_id` なし
- 予定したけど未実行: `planned_timeline_entries` だけ存在

という状態を表せます。

UIは、**1本の時間軸に plan と actual を重ねる**のが良いです。

```text
09:00 ┃ [Plan: 朝レビュー 09:00-09:30]
09:10 ┃   [Actual: 朝レビュー 09:10-09:35]
09:30 ┃
09:40 ┃ [Plan: 実装 09:40-11:00]
09:42 ┃   [Actual: 実装 09:42- ]
```

見た目としては、

- `plan`: 薄い色、枠線、背景
- `timeline`: 濃い色、塗りつぶし
- plan に紐づいた timeline は plan の上に重ねる
- timeline が進むほど plan が塗りつぶされる
- 予定外 timeline は通常の実績ブロックとして表示
- 未実行 plan は薄いまま
- 超過した actual は plan の外にはみ出して表示

イメージはこうです。

```text
09:00-10:00  Plan     [--------------------]
09:05-09:45  Actual     [############]
09:45-10:00  Remaining              [------]
```

状態は `status` だけで雑に持つより、実績から導出できるものは導出した方がいいです。

DBに持つなら最小限：

```ts
planned | cancelled
```

UI上の状態は計算で出せます。

- `not_started`: 紐づく timeline がない
- `in_progress`: 紐づく timeline の `end_at` が null
- `completed`: 紐づく timeline がある
- `overrun`: actual が planned_end_at を超えた
- `missed`: planned_end_at を過ぎても actual がない
- `cancelled`: plan 自体をキャンセル

開始予定・終了予定を持つなら、`daily_task_plans` ではなく `planned_timeline_entries` / `timeline_plans` みたいな名前が合います。  
個人的には既存の `timeline` と同列に扱いたいなら **`timeline_plans`** が一番わかりやすいです。

今の実装だと、timeline 側は `timeline_entries` です。

```sql
create table public.timeline_entries (
  id uuid primary key default gen_random_uuid(),
  task_id uuid not null references public.tasks(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  start_at timestamptz not null default now(),
  end_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  constraint timeline_entries_task_user_fk foreign key (task_id, user_id)
    references public.tasks(id, user_id) on delete cascade,

  constraint timeline_entries_end_after_start check (
    end_at is null or end_at >= start_at
  )
);
```

現在の意味づけはこうです。

- `tasks`: タスクのマスタ
- `timeline_entries`: 実績ログ
- `start_at`: 実際に開始した時刻
- `end_at`: 実際に終了した時刻。`null` なら実行中
- 実行中エントリは `end_at is null`
- 新しいタスクを開始すると、既存の実行中エントリを `end_at = now` で閉じてから、新しい `timeline_entries` を insert している

今の hook でも `ENTRY_SELECT` はこれだけです。

```ts
id,task_id,user_id,start_at,end_at,created_at,updated_at
```

なので、plan を重ねるなら一番自然なのは、`timeline_entries` はこのまま **actual / 実績** として維持して、別に `timeline_plans` を追加する形です。

```sql
create table public.timeline_plans (
  id uuid primary key default gen_random_uuid(),
  task_id uuid not null references public.tasks(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  planned_start_at timestamptz not null,
  planned_end_at timestamptz not null,
  status text not null default 'planned',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  constraint timeline_plans_task_user_fk foreign key (task_id, user_id)
    references public.tasks(id, user_id) on delete cascade,

  constraint timeline_plans_end_after_start check (
    planned_end_at > planned_start_at
  )
);
```

そして実績側に紐づけを足す。

```sql
alter table public.timeline_entries
add column timeline_plan_id uuid null references public.timeline_plans(id)
  on delete set null;
```

この設計だと、表示上は同じ時間軸に

- `timeline_plans`: 薄い予定ブロック
- `timeline_entries`: 濃い実績ブロック
- `timeline_entries.timeline_plan_id`: どの予定を塗りつぶした実績か

として扱えます。

既存の `timeline_entries` はかなりシンプルなので、ここに `planned_start_at` / `planned_end_at` を混ぜるより、`timeline_plans` を別テーブルにした方が「予定なしの突発実績」「予定したけど未実行」「予定と違う時間で実行」がきれいに表現できます。
